# 1 JUC锁体系
## 1.1 场景
JDK 1.5+ java.util.concurrent.locks 包核心组件，解决 synchronized 灵活性不足、粒度粗、无法中断/超时抢锁、读写场景性能低的问题，适用于需要可中断锁、公平锁、读写分离的高并发场景。

## 1.2 方案
基于 AQS 抽象队列同步器实现，分为独占锁（ReentrantLock/写锁）、共享锁（读锁）两类，通过 CAS 原子操作管理同步状态，CLH 队列存储等待线程，LockSupport.park/unpark 实现线程阻塞与唤醒；配套 Condition 组件实现精准线程等待-唤醒控制，解决 Object.wait()/notify() 唤醒粒度粗的问题。

## 1.3 实现
### 1.3.1 AQS 核心结构
1. 同步状态（state）：int 类型变量，通过 Unsafe.compareAndSwapInt 原子修改，不同锁对 state 语义定义不同；
2. CLH 等待队列：双向链表，节点包含线程引用、等待状态（CANCELLED/SIGNAL 等），抢锁失败线程封装为 Node 入队，按 FIFO 顺序等待；
3. 阻塞唤醒：依赖 LockSupport.park() 阻塞当前线程，LockSupport.unpark(Thread) 唤醒指定线程，park 响应中断，unpark 可提前调用且不丢失唤醒信号。

### 1.3.2 ReentrantLock（独占锁）实现
1. 加锁流程：
   - 非公平锁：直接调用 Unsafe.compareAndSwapInt(state, 0, 1) 抢锁，成功则设置当前线程为独占线程；失败则封装为 Node 入 CLH 队列，调用 LockSupport.park() 阻塞；
   - 公平锁：先检查 CLH 队列是否有等待节点，有则直接入队；无则执行 CAS 抢锁，失败入队并 park；
2. 解锁流程：
   - 调用 Unsafe.getAndAddInt(state, -1) 递减 state，若 state 归 0，获取 CLH 队列首节点，调用 LockSupport.unpark(首节点线程)；
   - 被唤醒线程重新执行 CAS 抢锁，成功则出队，失败则再次 park；
3. 重入逻辑：抢锁时先判断当前线程是否为独占线程，是则直接 state++，解锁时 state-- 至 0 才唤醒队列。

### 1.3.3 ReentrantReadWriteLock（读写锁）实现
1. state 位拆分：低 16 位表示写锁重入次数（独占），高 16 位表示读锁持有数（共享）；
2. 读锁加锁：调用 Unsafe.compareAndSwapInt(state, 原值, 原值 + (1 << 16)) 累加高 16 位，成功则持有读锁，失败入队 park；
3. 读锁解锁：调用 Unsafe.getAndAddInt(state, -(1 << 16)) 递减高 16 位，读锁数归 0 时唤醒队列；
4. 写锁加锁：先检查读锁数是否为 0，是则执行 Unsafe.compareAndSwapInt(state, 0, 1) 抢占低 16 位，成功则持有写锁；读锁数非 0 或 CAS 失败则入队 park；
5. 写锁解锁：调用 Unsafe.getAndAddInt(state, -1) 递减低 16 位，归 0 时唤醒队列。

### 1.3.4 Condition 实现
1. 核心结构：
   - 绑定关系：每个 Condition 实例由 Lock.newCondition() 创建，与 Lock 共享同一个 AQS state；
   - 等待队列：每个 Condition 维护独立的单向链表等待队列，节点结构与 AQS CLH 队列一致；
2. 核心操作：
   - await()：释放锁（CAS 重置 state 为 0）→ 线程入 Condition 等待队列 → park() 阻塞；
   - signal()：将 Condition 队列首节点移至 AQS CLH 队列 → unpark() 唤醒线程；
   - signalAll()：将 Condition 队列所有节点移至 AQS CLH 队列 → 依次 unpark() 唤醒；
3. 典型场景：生产者消费者模型，通过 notFull/notEmpty 两个 Condition 分别控制生产/消费线程的等待与唤醒。

## 1.4 实战
### 核心使用规范
| 组件类型                | 核心使用场景                 | 核心要点                          | 避坑点                              |
| :---------------------- | :--------------------------- | :-------------------------------- | :---------------------------------- |
| ReentrantLock（非公平） | 高并发独占资源访问           | 1. finally 中执行 unlock；2. 支持 tryLock 超时抢锁；3. 支持 lockInterruptibly 响应中断 | 1. 禁止漏解锁导致 state 无法归 0；2. 公平锁需排队，吞吐量低 |
| ReentrantLock（公平）   | 严格按请求顺序执行的场景     | 构造器传 true 开启，抢锁前检查队列 | 仅在必须保证执行顺序时使用          |
| ReentrantReadWriteLock | 读多写少（缓存/配置/热点数据） | 1. 读锁共享、写锁独占；2. 支持写→读锁降级，不支持读→写锁升级 | 1. 锁升级会导致死锁；2. 写锁长期持有阻塞所有读操作 |
| Condition               | 精准唤醒特定等待线程（生产者消费者/多条件等待） | 1. await()/signal() 调用前必须持有 Lock 锁；2. 多场景拆分多个 Condition 实现精准唤醒 | 1. 未持有锁调用抛 IllegalMonitorStateException；2. 避免所有线程共用一个 Condition |

# 2.底层原理
## 2.1 AQS 底层核心操作
1. CAS 操作：
   - 核心方法：Unsafe.compareAndSwapInt(Object obj, long stateOffset, int expect, int update)；
   - 作用：原子修改 state，保证多线程下状态修改的原子性，是抢锁的核心操作；
2. LockSupport 操作：
   - park()：阻塞当前线程，释放 CPU 资源，可响应中断，底层调用 pthread_cond_wait（Linux）/WaitForSingleObject（Windows）；
   - unpark(Thread thread)：唤醒指定线程，提前调用会使后续 park 直接返回，无阻塞；
3. CLH 队列操作：
   - 入队：CAS 修改队列尾节点，将新 Node 链接到队尾；
   - 出队：首节点抢锁成功后，CAS 修改队列头节点为下一个节点。

## 2.2 ReentrantLock 底层执行链路
1. 非公平锁加锁：
   CAS(state, 0, 1) → 成功（持有锁）→ 失败（入 CLH 队列 → park() 阻塞）；
2. 非公平锁解锁：
   state-- → state=0 → 获取 CLH 首节点 → unpark(首节点线程) → 首节点线程唤醒 → CAS(state, 0, 1) → 成功（出队）/失败（重新 park()）；
3. 重入：
   持有锁线程抢锁 → 不执行 CAS → state++ → 解锁时 state-- → 至 0 唤醒队列。

## 2.3 ReentrantReadWriteLock 底层执行链路
1. 读锁加锁：
   CAS(state, 原值, 原值 + (1 << 16)) → 成功（持有读锁）→ 失败（入队 → park()）；
2. 读锁解锁：
   state -= (1 << 16) → 读锁数=0 → 唤醒队列首节点；
3. 写锁加锁：
   读锁数=0 → CAS(state, 0, 1) → 成功（持有写锁）→ 失败（入队 → park()）；
4. 写锁解锁：
   state-- → state=0 → 唤醒队列首节点。

## 2.4 Condition 底层执行链路
1. await()：
   持有锁线程调用 await() → CAS(state, 持有值, 0) 释放锁 → 线程入 Condition 等待队列 → park() 阻塞；
2. signal()：
   持有锁线程调用 signal() → 将 Condition 队列首节点移至 AQS CLH 队列 → unpark(该节点线程) → 线程唤醒 → CAS 抢锁 → 成功（持有锁）/失败（park()）；
3. signalAll()：
   持有锁线程调用 signalAll() → Condition 队列所有节点移至 AQS CLH 队列 → 逐个 unpark() → 所有线程重新 CAS 抢锁。

## 2.5 死锁底层原因
1. 锁顺序相反：线程 A 持有锁1，抢锁2 CAS 失败 park()；线程 B 持有锁2，抢锁1 CAS 失败 park()，双方均无线程调用 unpark()；
2. 漏解锁：解锁时未执行 unlock，state 无法归 0，后续线程抢锁 CAS 失败，永久 park()；
3. 读锁升级写锁：读锁持有期间抢写锁，写锁 CAS 失败 park()，读锁未释放导致写锁永久阻塞；
4. Condition 误用：未持有锁调用 await()/signal()，抛异常导致锁未释放，线程永久 park()。

## 2.6 核心对比
| 操作         | AQS 实现方式                | synchronized 实现方式                     |
| :----------- | :-------------------------- | :---------------------------------------- |
| 抢锁         | CAS 修改 state + 队列排队   | monitorenter 指令，获取对象 monitor       |
| 阻塞         | LockSupport.park()          | Object.wait()，需先持有 monitor           |
| 唤醒         | LockSupport.unpark(指定线程) | Object.notify()，随机唤醒一个等待线程     |
| 精准唤醒     | Condition 多队列精准控制     | 无，仅能随机/全部唤醒                     |
| 中断响应     | park() 可直接响应中断       | wait() 响应中断，但需先获取 monitor       |

### 关键点回顾
1. AQS 核心是 state（CAS 管理）、CLH 队列（FIFO 等待）、park/unpark（阻塞唤醒），所有锁和 Condition 均基于此实现；
2. ReentrantLock 是 AQS 独占锁实现（state=重入次数），ReentrantReadWriteLock 拆分 state 高低位实现读写分离，Condition 是 AQS 扩展的多队列等待唤醒组件；
3. Condition 核心优势是多队列精准唤醒，await() 需先释放锁入专属队列，signal() 需将节点移回 AQS 队列才能抢锁；
4. 死锁根源是线程永久 park() 且无 unpark() 唤醒，核心规避手段是保证锁释放、锁顺序一致、合理使用 Condition。
