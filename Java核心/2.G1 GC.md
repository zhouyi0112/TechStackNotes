# 1.G1 GC 
## 1.1 场景
遵循**分代逻辑 + 分区物理 + 部分收集**核心设计理念，是 HotSpot 虚拟机官方主推的服务端垃圾回收器，为 JDK 9+ 默认回收器（出处：**JEP 248: Make G1 the Default Garbage Collector**）。
适用于多核服务器、4GB~64GB 中大型堆内存、要求**可预测低停顿**、规避内存碎片的业务场景。

## 1.2 方案
以固定大小 **Region** 作为堆内存最小管理单元
保留**新生代、老年代**逻辑分代规则；
采用**垃圾收益优先**的**部分收集**策略，通过**复制-整理**算法彻底避免内存碎片；
依托**停顿预测**模型，动态调控回收范围，严格保障 `-XX:MaxGCPauseMillis` 目标停顿时长（出处：**Java Virtual Machine Specification §11**）。

## 1.3 实现
1.  **Region 分区管理**：
    堆内存划分为 1MB~32MB（2的幂次）固定大小 Region
    分为 Eden、Survivor、Old、Humongous 四种类型。
2.  **RSet（Remembered Set）**：
    记录跨 Region 引用关系，支撑局部回收，无需全堆扫描，降低回收开销。
3.  **CSet（Collection Set）**：
    待回收 Region 集合，依据垃圾回收收益高低排序筛选。
4.  **标准回收流程**：
    初始标记(STW) → 并发标记 → 最终标记(STW) → 筛选回收(STW)。
5.  **GC 类型划分**：
    YoungGC（仅回收新生代）
    MixedGC（回收全新生代+部分老年代）
    FullGC（全堆串行回收，生产环境严禁出现）。

## 1.4 实战
### 核心参数配置（假设双核+1.5G堆场景）
| 参数配置                | 专业解释                                                                 | 双核+1.5G堆适配值 | 备注                     |
| :---------------------- | :----------------------------------------------------------------------- | ----------------- | :----------------------- |
| `-Xms<N> -Xmx<N>`       | 设定堆初始内存与最大内存为等值，禁止堆动态扩容，消除内存伸缩引发的 GC 波动 | `-Xms1536m -Xmx1536m` | 生产必配                 |
| `-XX:+UseG1GC`          | 启用 G1 垃圾回收器，JDK 8 需显式指定，JDK 9+ 默认启用                    | 必配              | 生产必配                 |
| `-XX:MaxGCPauseMillis=<N>` | GC 目标停顿软约束，G1 依据该值调控回收粒度，平衡延迟与吞吐量              | `200`             | 生产必配                 |
| `-XX:ParallelGCThreads=<N>` | STW 回收阶段并行线程数，决定 Stop-The-World 阶段回收效率                 | `2`（适配双核CPU）| 生产必配                 |
| `-XX:InitiatingHeapOccupancyPercent=<N>` | 触发并发标记与 MixedGC 的堆内存占用阈值，是防范 FullGC 的核心参数       | `40`              | 生产必配                 |
| `-XX:G1HeapRegionSize=<N>` | 设置 Region 分区大小，直接决定巨型对象的判定标准                        | `8m`              | 生产必配                 |
| `-XX:ConcGCThreads=<N>` | 并发标记阶段线程数，提升老年代垃圾标记效率，避免并发失败                 | `1`（适配双核CPU）| 生产必配                 |
| `-XX:+HeapDumpOnOutOfMemoryError` | OOM 时自动生成堆转储文件，用于内存泄漏排查                              | 必配              | 生产必配                 |
| `-XX:HeapDumpPath=<PATH>` | 堆转储文件存储路径                                                       | `./log`           | 生产必配                 |
| `-XX:+ExitOnOutOfMemoryError` | OOM 时强制退出进程，避免服务僵死                                        | 必配              | 生产必配                 |
| `-Xlog:gc*=info:file=<PATH>:time,uptime,level,tags` | JDK21 新版 GC 日志配置，记录完整回收信息                                | `./log/gc.log`    | 生产必配                 |
| `-XX:G1NewSizePercent=<N>` | 年轻代最小堆占比，限制新生代下限                                         | `5`（默认值）| 实验参数，一般不用（G1自适应更优） |
| `-XX:G1MaxNewSizePercent=<N>` | 年轻代最大堆占比，限制新生代上限                                         | `60`（默认值）| 实验参数，一般不用（易导致停顿抖动） |
| `-XX:G1MixedGCCountTarget=<N>` | 单次并发标记完成后，MixedGC 的执行次数                                   | `8`（默认值）| 实验参数，一般不用（G1自动调控） |
| `-XX:G1OldCSetRegionThresholdPercent=<N>` | 单次 MixedGC 回收老年代 Region 占比上限                                  | `10`（默认值）| 实验参数，一般不用（易引发停顿超时） |

# 2.底层原理
## 2.1 规范分层
- **JVM 规范（JVMS §11）**：定义垃圾回收基础语义、内存一致性约束，不绑定具体回收器实现。
- **HotSpot G1 设计规范**：基于分代内存语义，以 Region 分区、部分收集为核心，实现可预测停顿与低碎片回收。

## 2.2 核心组件原理
- **RSet**：每个 Region 独立维护，记录外部 Region 对当前 Region 的引用，回收时仅扫描 CSet 及关联 RSet，实现局部回收。
- **停顿预测模型**：基于历史 GC 耗时统计，动态计算本次可回收 Region 数量，确保单次停顿贴合目标值（出处：HotSpot G1 GC 源码注释）。

## 2.3 GC 触发规则
- **YoungGC**：Eden Region 空间耗尽触发。
- **MixedGC**：堆内存占用达到 IHOP 阈值触发（适配值 40%，提前触发规避 FullGC）。
- **FullGC**：堆内存耗尽、并发标记失败、巨型对象密集分配触发（生产级故障，需紧急排查）。

# 3 生产环境实战脚本
## 3.1 启动脚本（start-server.sh）
```bash
#!/bin/bash
jar_name=打包文件.jar
work_dir=工作目录
log_dir=$work_dir/log
pid_file=$work_dir/server.pid

# 前置准备：创建日志目录
mkdir -p $log_dir

# 停止旧进程，避免端口/进程冲突
if [ -f $pid_file ]; then
  pid=$(cat $pid_file)
  ps -p $pid >/dev/null 2>&1 && kill $pid && sleep 3
  ps -p $pid >/dev/null 2>&1 && kill -9 $pid
  rm -f $pid_file
fi

# 启动服务（嵌入G1 GC核心调优参数，实验参数仅作示例标注，生产默认不启用）
cd $work_dir || exit 1

nohup java \
-Xms1536m \
-Xmx1536m \
-XX:+UseG1GC \
-XX:MaxGCPauseMillis=200 \
-XX:ParallelGCThreads=2 \
-XX:InitiatingHeapOccupancyPercent=40 \
-XX:G1HeapRegionSize=8m \
-XX:ConcGCThreads=1 \
# 以下为实验参数，生产环境一般不用，仅作认知标注
# -XX:G1NewSizePercent=5 \
# -XX:G1MaxNewSizePercent=60 \
# -XX:G1MixedGCCountTarget=8 \
# -XX:G1OldCSetRegionThresholdPercent=10 \
-XX:+HeapDumpOnOutOfMemoryError \
-XX:HeapDumpPath=$log_dir \
-XX:+ExitOnOutOfMemoryError \
-Xlog:gc*=info:file=$log_dir/gc.log:time,uptime,level,tags \
-Xlog:gc+heap=debug \
-XX:+UnlockDiagnosticVMOptions \
-XX:+LogVMOutput \
-XX:LogFile=$log_dir/jvm.log \
-jar $jar_name \
-Dspring.profiles.active=prod \
> /dev/null 2>&1 &

# 记录进程PID并校验启动结果
echo $! > $pid_file
sleep 3

if ps -p $(cat $pid_file) >/dev/null 2>&1; then
  echo "✅ 服务启动成功 PID: $(cat $pid_file)"
  echo "📊 GC日志路径: $log_dir/gc.log"
  echo "📄 业务日志路径: $log_dir/app.log"
  echo "🔍 JVM诊断日志: $log_dir/jvm.log"
  echo "💡 注：脚本中实验性GC参数已注释，生产环境默认依赖G1自适应策略"
else
  echo "❌ 服务启动失败，请检查日志排查问题"
  exit 1
fi
```

## 3.2 实验参数补充说明
| 实验参数                                   | 核心作用                     | 不推荐使用的原因                               |
| :------------------------------------- | :----------------------- | :------------------------------------- |
| `G1NewSizePercent/G1MaxNewSizePercent` | 固定年轻代内存占比上下限             | G1 原生自适应算法可根据停顿目标动态调整新生代大小，手动固定易导致停顿抖动 |
| `G1MixedGCCountTarget`                 | 限定单次并发标记后MixedGC执行次数     | 手动限制易导致老年代回收不彻底或过度回收，G1自动调控更贴合实际内存状态   |
| `G1OldCSetRegionThresholdPercent`      | 限制单次MixedGC回收老年代Region占比 | 调大易引发停顿超时，调小易导致老年代堆积，默认值已适配多数场景        |
