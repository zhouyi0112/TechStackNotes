# 1.JUC 常用同步协作工具
## 1.1 场景
JUC 提供的多线程协作同步工具，用于解决线程间等待、限流、屏障同步等时序协调问题，替代手动锁+等待唤醒的实现。

## 1.2 方案
基于 AQS（CountDownLatch/Semaphore）或 Lock+Condition（CyclicBarrier）实现，通过 CAS 原子修改状态值，结合 park/unpark 实现线程阻塞与唤醒，适配不同协作场景的语义。

## 1.3 实现
### 1.3.1 CountDownLatch（倒计时门）
1. 核心逻辑：**一次性计数器**，初始化指定计数，线程调用 `countDown()` 递减计数，调用 `await()` 的线程阻塞等待计数归 0。
2. 关键特性：计数只能递减、不可重置，一旦归 0 永久生效。
3. 核心方法：
   - `CountDownLatch(int count)`：初始化计数（count > 0）；
   - `countDown()`：计数-1，CAS 原子操作；
   - `await()`：阻塞至计数=0（可设置超时时间）。

### 1.3.2 CyclicBarrier（循环屏障）
1. 核心逻辑：**可循环复用的屏障**，一组线程互相等待，直到所有线程到达屏障点（计数达标），触发屏障任务后重置计数，可重复使用。
2. 关键特性：区别于 CountDownLatch 的“一次性”，支持循环；可自定义屏障任务（所有线程达标后执行）。
3. 核心方法：
   - `CyclicBarrier(int parties)`：初始化参与屏障的线程数；
   - `CyclicBarrier(int parties, Runnable barrierAction)`：额外指定屏障触发后执行的任务；
   - `await()`：线程到达屏障点，阻塞等待其他线程，达标后唤醒所有线程并执行屏障任务。

### 1.3.3 Semaphore（信号量）
1. 核心逻辑：**许可控制**，通过“许可数”限制同时访问资源的线程数，线程 `acquire()` 获取许可（许可数-1），`release()` 释放许可（许可数+1）。
2. 关键特性：支持公平/非公平模式，许可数可动态调整（release() 可超量释放）。
3. 核心方法：
   - `Semaphore(int permits)`：初始化许可数（默认非公平）；
   - `Semaphore(int permits, boolean fair)`：指定公平/非公平模式；
   - `acquire()`：获取1个许可，无许可则阻塞；
   - `release()`：释放1个许可。

## 1.4 实战
| 工具类          | 核心场景                          | 核心注意事项                          |
|-----------------|-----------------------------------|---------------------------------------|
| CountDownLatch  | 主线程等待多任务完成、多线程同步启动 | 计数一次性，不可重置；await()建议加超时 |
| CyclicBarrier   | 一组线程互相等待后统一执行（如分批处理） | 线程中断会触发 BrokenBarrierException；可循环使用 |
| Semaphore       | 限流（如接口并发数、池资源控制）| 非公平模式性能更高；避免超量 release() |

# 2.底层原理
## 2.1 CountDownLatch 底层（基于 AQS 共享模式）
1. AQS 状态（state）：对应初始化的计数，默认非公平；
2. `countDown()`：调用 AQS 的 `releaseShared()`，CAS 原子递减 state，直到 state=0；
3. `await()`：调用 AQS 的 `acquireShared()`，自旋检查 state 是否为 0，非 0 则 park 阻塞；
4. 唤醒逻辑：当 state 减至 0 时，触发 AQS 共享模式的唤醒逻辑，unpark 所有等待线程。

## 2.2 CyclicBarrier 底层（基于 ReentrantLock + Condition）
1. 核心组件：
   - `ReentrantLock`：保证计数修改的线程安全；
   - `Condition`：维护等待线程的条件队列；
   - `count`：当前等待的线程数（初始=parties）；
2. `await()` 逻辑：
   加锁 → count-1 → 若 count≠0，Condition.await() 阻塞 → 若 count=0，执行屏障任务 → 重置 count → Condition.signalAll() 唤醒所有线程 → 解锁；
3. 循环复用：计数重置后，可重复触发屏障逻辑。

## 2.3 Semaphore 底层（基于 AQS 共享模式）
1. AQS 状态（state）：对应许可数；
2. `acquire()`：
   公平模式：检查队列是否有等待线程，有则入队；无则 CAS 递减 state，不足则入队 park；
   非公平模式：直接 CAS 递减 state，失败则入队 park；
3. `release()`：CAS 递增 state，触发 AQS 唤醒逻辑，unpark 队列中等待的线程。

## 2.4 常见问题
1. CountDownLatch：await() 未设置超时，若某线程未执行 countDown()，会导致等待线程永久阻塞；
2. CyclicBarrier：任意线程中断/超时，会标记屏障“破损”，后续线程 await() 直接抛 BrokenBarrierException；
3. Semaphore：release() 次数超过 acquire()，会导致许可数无限增加，失去限流意义；公平模式下性能显著降低。
