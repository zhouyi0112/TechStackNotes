# 1. JMM(Java内存模型)

## 1.1 场景
多线程无同步访问共享变量，会出现**可见性、有序性、原子性**三大问题，产生 **Data Race**，导致程序结果不可预测。

## 1.2 方案
Java 提供 **JMM 内存模型（JSR‑133/JLS §17）**，核心是 **Happens‑Before（HB）规则**：
若操作A Happens‑Before操作B，则A的结果对B可见，且执行顺序优先于B。

> 官方依据：
> - JLS §17.4.5 Happens-before Order

## 1.3 实现（触发HB的核心机制）
1. **程序顺序规则（Program Order Rule）**
   单线程中，按照程序语义，先执行的操作 HB 后执行的操作。
   ✅ 案例：单线程内执行 `int a=1; int b=a+1;`，JMM保证b必为2，不会因指令重排导致b=0。

2. **监视器锁规则（Monitor Lock Rule）**
   对一个监视器锁的解锁操作（unlock） HB 后续对同一监视器锁的加锁操作（lock）。
   ✅ 案例：线程1通过`synchronized(lock){count=10;}`解锁后，线程2在同一锁内必能读到count=10。

3. **volatile变量规则（Volatile Variable Rule）**
   对volatile变量的写操作 HB 后续对该volatile变量的读操作。
   ✅ 案例：线程1写`volatile flag=1`，线程2读flag最终能看到1，不会一直读到初始值0。

4. **线程启动规则（Thread Start Rule）**
   调用线程A的`Thread.start()`操作 HB 线程A内的所有操作。
   ✅ 案例：主线程设置`num=5`后调用`threadA.start()`，线程A内必能读到num=5。

5. **线程终止规则（Thread Termination Rule）**
   线程A内的所有操作 HB 检测到线程A终止的操作（如`Thread.join()`返回、`Thread.isAlive()`返回false）。
   ✅ 案例：主线程调用`threadA.join()`后，必能看到线程A对共享变量的所有修改。

6. **中断规则（Interruption Rule）**
   线程A调用线程B的`interrupt()`操作 HB 线程B检测到中断（如`Thread.interrupted()`/`isInterrupted()`返回true）。
   ✅ 案例：线程A调用`threadB.interrupt()`，线程B后续调用`isInterrupted()`必能检测到中断信号。

7. **终结器规则（Finalizer Rule）**
   对象的构造函数执行完成 HB 该对象的终结器（finalize()）执行。
   ✅ 案例：对象`new Object()`构造完成后，JVM才会调用其`finalize()`方法，避免构造未完成就执行终结逻辑。

8. **传递性规则（Transitivity Rule）**
   若 A HB B 且 B HB C，则 A HB C。
   ✅ 案例：A(写volatile变量) HB B(解锁)，B HB C(加同一锁) → A HB C，C能看到A的volatile变量修改。

> 官方依据：JLS §17.4.5

## 1.4 实战
- **volatile**：仅保证可见性、有序性，**不保证原子性**
- **synchronized/Lock**：同时保证原子性、可见性、有序性
- 无同步机制：JMM 不提供任何一致性保证

# 2. 底层原理

## 2.1 规范分层：JVM规范 vs JMM规范
- **JVM规范（JVMS §6.5）**：定义标准字节码指令集，包括`monitorenter`、`monitorexit`，强制所有合规JVM（HotSpot、OpenJ9等）实现，仅规定指令**语义**，不限制底层实现
- **JMM规范（JLS §17）**：定义多线程内存一致性规则，核心是**Happens‑Before偏序关系**，规定`lock/unlock`动作的HB约束

> 官方依据：
> - JVMS §6.5 Instructions（https://docs.oracle.com/javase/specs/jvms/se21/html/jvms-6.html#jvms-6.5）
> - JLS §17.4 Memory Model（https://docs.oracle.com/javase/specs/jls/se21/html/jls-17.html）

## 2.2 指令语义与JMM的官方联动
- `monitorenter` 指令语义（JVM规范）：获取对象监视器锁，对应JMM的**lock action**
- `monitorexit` 指令语义（JVM规范）：释放对象监视器锁，对应JMM的**unlock action**
- 核心逻辑：因指令语义与JMM锁动作严格对应，**实现指令语义即自动满足JMM的HB锁规则**（unlock HB 后续lock）

> 官方依据：JVMS §6.5（monitorenter/monitorexit） + JLS §17.4.5（Monitor Lock Rule）

## 2.3 不同JVM的实现自由
- HotSpot：基于`ObjectMonitor`实现，支持偏向锁、轻量级锁、重量级锁等优化机制
- Eclipse OpenJ9：基于`J9Monitor`独立实现同步逻辑，底层机制与HotSpot不同
- 其他合规JVM：可采用OS原生锁、CAS等任意底层方式，**仅需保证指令语义不变**，即可符合JMM规范

> 官方依据：JVMS §2.11.10 Synchronization

## 2.4 Synchronized关键字
- 语法层：`synchronized` 是Java同步关键字，经`javac`编译后会转化为`monitorenter`（抢锁）和`monitorexit`（释放锁）字节码指令（异常路径会额外生成`monitorexit`保证锁释放）；
- 规范层：`monitorenter/monitorexit` 的语义对应JMM的`lock/unlock`动作，因此满足HB规则（同一锁的unlock HB后续lock）；
- 效果层：最终实现原子性、可见性、有序性。

> 官方依据：
> - JVMS §6.5 monitorenter（https://docs.oracle.com/javase/specs/jvms/se21/html/jvms-6.html#jvms-6.5.monitorenter）
> - JVMS §6.5 monitorexit（https://docs.oracle.com/javase/specs/jvms/se21/html/jvms-6.html#jvms-6.5.monitorexit）
