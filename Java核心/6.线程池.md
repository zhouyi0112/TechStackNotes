# 1.线程池
## 1.1 场景
JUC 提供的线程复用与任务调度工具，用于高并发场景下统一管理线程，避免频繁创建销毁线程，控制并发线程数量，保护系统资源。

## 1.2 方案
采用核心线程、阻塞队列、最大线程、拒绝策略四层调度模型
基于生产者-消费者模式实现任务异步执行
底层依靠 Lock、Condition、park/unpark 实现线程等待与唤醒。

## 1.3 实现
### 1.3.1 核心 7 参数
1. corePoolSize：核心线程数，常驻线程，默认不会被回收
2. maximumPoolSize：最大线程数，包括核心线程 + 非核心线程
3. keepAliveTime：非核心线程空闲超时时间，超时则销毁
4. unit：keepAliveTime 的时间单位
5. workQueue：阻塞队列，用于缓存等待执行的任务
6. threadFactory：线程工厂，用于创建线程，可自定义名称、优先级
7. handler：拒绝策略，任务达到上限时的处理策略

### 1.3.2 任务执行流程
提交任务 → 核心线程数未满 → 创建核心线程执行
核心线程已满 → 任务加入阻塞队列
队列已满 → 创建非核心线程执行
线程数达到 maximumPoolSize → 执行拒绝策略

### 1.3.3 4 种拒绝策略
1. AbortPolicy
    默认策略，直接抛出 RejectedExecutionException 异常。
2. CallerRunsPolicy
    由提交任务的线程自己执行该任务，不抛异常、不丢任务，生产环境最稳定。
3. DiscardPolicy
    直接静默丢弃当前任务，不抛异常，无感知。
4. DiscardOldestPolicy
    丢弃队列中等待最久的任务，然后再次尝试提交当前任务。

## 1.4 实战
CPU 密集型：corePoolSize = CPU 核心数
IO 密集型：corePoolSize = CPU 核心数 * 2
生产环境必须使用**有界队列**，防止无界队列导致 OOM
拒绝策略优先使用 CallerRunsPolicy
禁止使用 Executors 创建线程池，必须手动 new ThreadPoolExecutor

# 2.底层原理
## 2.1 核心结构
### 2.1.1 状态与线程数的复合存储
线程池内部用一个 **32位 int 变量（ctl）** 同时存储两个核心信息：
- 高 3 位：线程池运行状态（RUNNING、SHUTDOWN、STOP、TIDYING、TERMINATED）
- 低 29 位：当前工作线程数
  这样设计的核心目的是**通过一次 CAS 操作同时保证状态和线程数的原子性修改**，避免多线程下的不一致问题。

#### 线程池5种运行状态详解：
| 状态       | 核心含义                                                                 |
|------------|--------------------------------------------------------------------------|
| RUNNING    | 正常运行，可接收新任务、处理队列任务                                     |
| SHUTDOWN   | 调用 shutdown()，不接收新任务，但处理队列剩余任务                         |
| STOP       | 调用 shutdownNow()，不接收新任务、不处理队列任务，中断所有正在执行的任务   |
| TIDYING    | 「整理/就绪中」，临时过渡状态：<br>1. 触发条件：SHUTDOWN状态下队列空且线程数为0 / STOP状态下线程数为0；<br>2. 核心行为：执行 terminated() 钩子方法（默认空实现，可自定义收尾逻辑）；<br>3. 作用：保证线程池终止前完成收尾工作 |
| TERMINATED | terminated() 执行完成，线程池完全终止                                   |

### 2.1.2 Worker 工作线程封装
线程池中的所有工作线程都被封装为 **Worker 类**，核心设计：
1. 继承 AQS 并实现**独占锁**：
   - 锁的作用：标记线程是否正在执行任务，避免线程在执行任务时被中断（仅空闲线程可被中断）；
   - 实现方式：重写 AQS 的 tryAcquire() 方法，用一个 boolean 变量标记锁状态（0=未锁定，1=锁定）。
2. 实现 Runnable 接口：
   - Worker 本身是可执行的任务，启动后会进入**无限循环**：从阻塞队列中获取任务并执行；
   - 核心逻辑：`getTask()` 方法从队列取任务，取到则执行，取不到则退出循环并销毁线程。
3. 线程复用核心：
   Worker 线程启动后不会单次执行完任务就销毁，而是循环从队列取任务，直到线程池状态变更或超时，实现线程复用。

## 2.2 调度原理
线程启动后循环从阻塞队列获取任务
队列为空时，线程在 Condition 上等待，调用 park 阻塞
新任务入队后，signal 唤醒空闲线程，unpark 继续执行
非核心线程超时未获取到任务，自动退出销毁

## 2.3 底层操作
线程安全控制：CAS + ReentrantLock
线程阻塞与唤醒：Condition.await() / signal()
底层依赖：LockSupport.park() / unpark()

## 2.4 常见问题
无界队列导致内存溢出 OOM
线程数设置不合理导致 CPU 过高或吞吐量过低
任务中抛出未捕获异常，会导致当前线程被销毁重建
线程池未优雅关闭，导致应用无法退出或资源泄漏
